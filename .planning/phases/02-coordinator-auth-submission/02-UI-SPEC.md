# Phase 2 — UI Design Contract

> Visual and interaction contract for Phase 2 coordinator surfaces: auth pages (`/login`, `/register`, `/verify-email`), onboarding profile form (`/onboarding`), coordinator dashboard (`/dashboard`), BIP submission wizard (5 steps), and session-aware public `StickyNav`. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Phase 2 visual surfaces in scope: `(auth)` route group layout + login/register/verify-email pages, `(dashboard)` route group layout + nav, `/onboarding`, `/dashboard`, BIP submission wizard steps 1–5, two-tab conflict modal, session expiry recovery, save-status indicator, public StickyNav session-aware variant. **Out of scope:** admin panel, email template HTML, GDPR consent, `/what-is-a-bip`.

**Phase 2 exceptions (pre-approved):**

- **Auth card width:** max-w-[440px] (440px) — fits comfortably on 375px iPhone with 16px side padding. Narrower than Phase 1 content cards; this is intentional for focused auth UX.
- **Auth card padding:** 40px all sides (closest 4-multiple: `p-10`). Matches Phase 1 card pattern at `shadow-md`; lighter than the 48px UniversityCTA card.
- **Wizard card:** full-page-width within the `(dashboard)` container on mobile; max-w-[760px] centered on desktop. Not a narrow auth card.
- **Dashboard container:** same 1200px max-w as Phase 1 public pages, 24px side padding desktop, 16px mobile.
- **Tab bar height:** 48px (matches touch target minimum, aligns with Phase 1 nav height rhythm).
- **Status badge vertical padding:** 4px top/bottom (`py-1`), 8px left/right (`px-2`). Use `py-1 px-2` in Tailwind.
- **Wizard step indicator circle:** 32px diameter — smaller than the Phase 1 "How it works" 48px circles; these are functional navigation, not decorative.
- **Save-status indicator height:** 24px (inline text element, not a card or bar).

## Auth Page Layout Contract (D-12, D-13)

These rules govern the three auth pages (`/login`, `/register`, `/verify-email`) exclusively.

- **Page background:** `bg-bg-soft` (`#f7f8fc`) — full-screen, no public nav or footer.
- **Layout:** `min-h-screen grid place-items-center px-4` — vertically and horizontally centered.
- **Card:** `bg-white rounded-lg shadow-md p-10 w-full max-w-[440px]`. One card per page. No sidebar.
- **Card header:** BipHub `<LogoMark>` (32px) + wordmark centered at card top, 24px below the logo: page h3 heading. No eyebrow labels on auth cards — these are functional pages.
- **Card footer:** Single line of 14px/400/muted text: contextual link ("Already have an account? Sign in" on register; "No account? Register" on login). 16px below the primary submit button.
- **No StickyNav, no public Footer on auth pages.** The `(auth)/layout.tsx` renders no chrome beyond the centered layout wrapper.
- **Legal line:** 11px / muted / centered at page bottom (outside card, fixed to page bottom): `Independent project — not affiliated with the European Commission`. This satisfies INFO-03 on auth pages where the public Footer is absent.

## Public StickyNav Session-Aware Contract (D-15)

When a coordinator session is detected, the public `StickyNav` right-side slot transforms:

- **Logged-out state (Phase 1 current):** `Sign in` (ghost) + `List your BIP` (primary).
- **Logged-in state (Phase 2 addition):** Remove both auth CTAs. Replace with:
  - `Dashboard` text link (14px / 600 / `text-ink` → hover `text-eu-blue`)
  - Coordinator initials avatar circle (32px) — same as dashboard nav above.
- **Implementation:** The `(public)/layout.tsx` RSC passes `hasClaims: boolean` + `initials: string | null` as props to `<StickyNav>`. No client-side flash — session is checked at RSC render time via `getClaims()`. If `getClaims()` returns null, show logged-out state.

## Onboarding Page Contract (`/onboarding`)

Single-page form — not a multi-step wizard. All fields visible at once, grouped into two sections.

- **Layout:** Same `(dashboard)` route group chrome (nav bar) but content is a centered card: `bg-white rounded-lg shadow-md p-10 max-w-[560px] mx-auto my-12`.
- **Page h3:** `Complete your profile` (22px / 600).
- **Sub-copy:** `Before you can submit a BIP, we need a few details about you and your university. This takes about 2 minutes.` (14px / 400 / muted).
- **Section 1 — Personal details:** `Full name` (text input), `Contact email` (email input, pre-filled with auth email, editable).
- **Section 2 — University details:** `Your university` (combobox — searchable from `universities` table; inline "Add new university" option at list bottom if not found), `Country` (select, locked to `lib/countries.ts`), `Erasmus code` (text input, placeholder `ABC UNI01`).
- **Submit button:** Full-width, `bg-eu-blue text-white rounded-pill` — label: `Complete profile →`.
- **Validation:** All 5 fields required. React Hook Form + Zod v3. Inline field-level errors below each input (14px / `text-status-rejected`).
- **"Add new university" inline flow:** When user types a name not in the autocomplete list, a secondary card slides in below the combobox: `New university name` (pre-filled from search query) + `Country` select. A `Add university` button submits the new row and selects it in the combobox. This flow stays inside the onboarding form — no modal, no route change.

## Wizard Layout Contract (`BipSubmissionWizard`)

The wizard is a `'use client'` component mounted inside `app/(dashboard)/dashboard/bips/new/page.tsx` and `app/(dashboard)/dashboard/bips/[id]/edit/page.tsx`.

### Outer shell

- **Page background:** `bg-bg-soft`.
- **Wizard card:** `bg-white rounded-lg shadow-md w-full max-w-[760px] mx-auto my-8`.
- **Wizard header (sticky within card):** `border-b border-border px-8 py-4 flex items-center justify-between`.
  - Left: `Step {n} of 5` in body-sm style (14px / 400 / muted) + current step title in body-sm (14px / 600 / ink).
  - Center (or right of title): Step indicator dots — 5 circles, 8px diameter, 8px gap. Active: `bg-eu-blue ring-2 ring-eu-gold ring-offset-2`. Completed: `bg-eu-blue`. Upcoming: `bg-border`.
  - Right: Save-status indicator — see contract below.
- **Wizard body:** `px-8 py-6` (or `px-6 py-5` on mobile). Max-height `calc(100vh - 180px)`, `overflow-y-auto`.
- **Wizard footer (sticky):** `border-t border-border px-8 py-4 flex items-center justify-between`.
  - Left: `← Back` ghost button (hidden on Step 1).
  - Right: `Save & Continue →` primary button (Steps 1–4) / `Submit for review →` gold primary button (Step 5 Preview).

### Save-status indicator (D-02)

Positioned in the wizard header, right side, 14px / 400 / muted. Three states only — implement as a single text element, no icons needed (icon optional: spinner on saving):

| State | Text | Color |
|-------|------|-------|
| Idle (after successful save) | `Saved` | `text-muted` |
| Saving (action in flight) | `Saving…` | `text-muted` |
| Failed (D-03) | See failure UI below | — |

**Save failure UI (D-03):** Replace save-status text with: `Save failed — ` + `Retry` as an inline text-button (`text-eu-blue underline cursor-pointer`). Simultaneously, show a shadcn Sonner toast: `Failed to save draft. Your changes are preserved — tap Retry to try again.` (5s duration, no auto-dismiss on error). The Zustand store retains all form data; no input is lost.

### Two-tab conflict modal (D-04)

shadcn `Dialog` — not a toast, not a banner. Triggered when `saveDraftAction` returns a stale `updated_at` conflict.

- **Title:** `Draft updated in another tab`
- **Body:** `Your draft was changed in another browser tab. What would you like to do?`
- **Buttons (stacked, full-width):**
  1. `Reload to get latest` — primary blue — discards current wizard state, fetches fresh draft, reloads wizard from Step 1.
  2. `Overwrite with this version` — destructive variant (red text, border) — force-saves current wizard state over the other tab's version.
- **No auto-dismiss.** User must choose. No close (X) button.

## Wizard Steps — Field and Layout Contracts

### Step 1 — Basic Info

Fields: `title` (text input), `isced_f_code` (select — 8 ISCED groups from `lib/isced.ts`), `description` (textarea, 4 rows min), `learning_outcomes` (textarea, 3 rows min — hint text: `One outcome per line, or a free-text paragraph`).

- **Section label above field group:** none — fields have inline labels. No section headings inside steps (they are the h3 step title).
- **Character count** shown below `title` input: `{n}/120 characters` (14px / 400 / muted). No hard limit in the UI — Zod enforces 120 max server-side.
- **Field order:** title → field of study → description → learning outcomes.

### Step 2 — Program Details

Fields: `virtual_component_description` (textarea, 3 rows), `virtual_timing` (select: `Before physical mobility` / `After physical mobility` / `Concurrent`), `host_city` (text input), `physical_start_date` (date input — use `<input type="date">` not shadcn Calendar to avoid calendar popup complexity; Calendar reserved for the public `/bips` filter), `physical_end_date` (date input), `application_deadline` (date input), `ects_credits` (number input, min 1 max 30, step 1), `max_participants` (number input, min 5 max 20), `study_levels` (checkbox group: Bachelor / Master / PhD — at least one required), `language_of_instruction` (select — ISO 639-1 language list, prioritize EU languages), `language_level_min` (select — CEFR: A1 / A2 / B1 / B2 / C1 / C2 / None required).

- **Date field hint:** `YYYY-MM-DD format` shown as placeholder text.
- **Dates validation note in UI:** below the `application_deadline` field, inline small text: `Deadline must be before the physical start date.` (shown always, not just on error).
- **Participants range note:** `Erasmus+ allows 5–20 funded participants.` (help text, always visible below the input).

### Step 3 — Partners

Fields: `host_university` (read-only — pre-filled from coordinator's profile `university_id`, shown as a gray locked input with the university name; coordinator cannot change host university here), `partner_universities` (multi-entry combobox — same university combobox as onboarding, but multi-select; each selected university appears as a removable chip below the combobox).

- **Host university locked state:** `bg-bg-soft text-ink-2 cursor-not-allowed` input appearance. Caption below: `Host university is set from your profile. To change it, update your profile settings.` (but no link to profile settings in Phase 2 — caption only).
- **Partner chips:** `bg-eu-blue-50 text-eu-blue text-sm rounded-pill px-3 py-1` with an ×  dismiss icon on the right. Partners added via combobox; also supports free-text entry (user types a name not in the list → chip is created with a `(unverified)` suffix appended by the Server Action, matching the public BIP detail page display from Phase 1).
- **Empty state caption (no partners yet):** `No partner universities added yet. At least one partner is recommended for Erasmus+ BIPs.` (not a hard validation error — coordinators can submit without partners).

### Step 4 — Application Info

Fields: `green_travel` (Switch toggle), `inclusion_support` (Switch toggle), `eligibility_notes` (textarea, 3 rows, optional — placeholder: `Describe any language requirements, academic prerequisites, or open access policy`), `how_to_apply_type` (radio group: `External URL` / `Coordinator contact`), `how_to_apply_value` — conditionally rendered:
  - If URL: `url` text input (placeholder: `https://...`).
  - If contact: `contact_name` text input + `contact_email` email input (two fields).

- **Switch toggle layout:** Switch on the left, label to the right (14px / 600 / ink), caption below (14px / 400 / muted). Green travel caption: `Coordinators offering green travel support receive Erasmus+ bonus funding.` Inclusion support caption: `Indicates the BIP accommodates students with disabilities or special needs.`
- **How-to-apply radio group label:** `How do students apply?` (14px / 600).
- **Conditional field reveal:** The URL/contact fields animate in with `LazyMotion` `m.div` opacity + height transition (200ms / ease) when the radio selection changes.

### Step 5 — Preview

No form fields. Read-only render of the full BIP detail layout.

- **Layout:** Renders `<BipBody>` and `<BipSidebar>` using wizard Zustand store data, identical to the public `/bip/[slug]` page layout — this is a live preview of how the published BIP will appear.
- **Preview banner (top of step):** `bg-eu-blue-50 border border-eu-blue-100 rounded-md px-4 py-3 text-sm text-eu-blue` with text: `This is a preview of your BIP. It won't be visible publicly until reviewed and approved by the BipHub team.`
- **Submit button in wizard footer:** `Submit for review →` — gold variant (`bg-eu-gold text-ink hover:bg-eu-gold-dark rounded-pill`). This matches the D-11 "prominent gold CTA" pattern. Clicking calls `submitBipAction` which sets `status = 'pending'`.
- **Post-submit redirect:** On `submitBipAction` success → redirect to `/dashboard?submitted=true`. The dashboard renders a Sonner toast: `Your BIP has been submitted for review. We'll notify you by email once it's been reviewed.` (5s, no auto-dismiss on success).

## Dashboard Page Contract

### Layout

- **Page header:** Full-width row, `bg-white border-b border-border px-6 py-5`.
  - Left: `h3` `Your BIPs` (22px / 600 / ink) + sub-line `{n} BIPs total` (14px / 400 / muted).

Active tab: `border-b-2 border-eu-blue text-ink font-semibold`. Count badge on each tab (except `All` which shows total): `bg-border text-ink-2 text-xs font-semibold rounded-full px-2 py-1 ml-2`.
- **BIP list:** Below the tab bar, full-width card list (not a grid — this is a management view). Each item is a `<DashboardBipCard>` — see below.

- **Empty state (filtered tab, e.g. 0 Rejected BIPs):** Minimal: `No {status} BIPs.` (muted, 14px, centered). No CTA needed.

### DashboardBipCard

A list-style card (not the public BipCard grid variant). Same `bg-white border border-border rounded-lg shadow-sm` container.

- **Layout:** Single row on desktop — left 60%: title (16px / 600) + sub-line (university · host city) + ISCED field tag chip. Right 40%: status badge + date line + action buttons.
- **Status badge:** Pill chip using the status color tokens: `bg-{status}-bg text-{status} border border-{status} rounded-pill text-xs px-2 py-1 font-semibold`.
- **Date line:** `Submitted {date}` or `Last saved {date}` — 14px / 400 / muted.
- **Action buttons (D-10 — per status):**
  - **Draft:** `Edit` (ghost button, 14px) + `Delete` (destructive ghost, 14px / `text-status-rejected`).
  - **Pending:** `Edit` (ghost) + `Withdraw` (ghost / `text-status-pending` — conveys cautious action, not destructive).
  - **Approved:** `View public page →` (text link, 14px / eu-blue).
  - **Rejected:** `View details` (text link) — the rejection reason is shown inline below the card title as a callout (see below).
- **Rejection reason callout (rejected BIPs only):** Inline, directly below the title inside the card. `border-l-4 border-eu-gold pl-3 py-1 bg-eu-gold-soft rounded-r-sm` — label: `Reason: ` (14px / 600 / ink-2) + reason text (14px / 400 / ink-2). Not a modal; always visible on card.
- **`is_seed` pill:** NOT shown on dashboard BipCards (from 02-CONTEXT.md "Specific Ideas" note — coordinators submit real BIPs, `is_seed` is a public browsing detail only).
- **Hover state:** `border-border-strong shadow-sm` — subtle, no translateY lift (this is a list management view, not a discovery card).
- **Mobile (< 960px):** Stack layout — title + sub-line + field tag, then status badge + date on same row, then action buttons as full-width row below.

## Interaction Contracts

### Forms (all auth + onboarding + wizard steps)

- **Validation trigger:** `onBlur` per field for inline error messages. `onSubmit` for full-form validation. Do NOT validate on every keystroke (typing `ema` should not show "invalid email").
- **Error message position:** Immediately below the input, 4px gap. 14px / 400 / `text-status-rejected`. One error per field.
- **Success state:** No green checkmark per field — absence of error is sufficient. The save-status indicator in the wizard header covers the save success state.
- **Disabled state:** Submit button disabled while a Server Action is in flight (`isPending` from `useActionState` or manual `useState`). Button shows spinner (Lucide `Loader2` rotating, 16px, same color as button text).
- **Autofocus:** On auth pages, autofocus the first input field on mount. On wizard steps, autofocus the first field when the step renders.
- **Autocomplete attribute:** Login email: `autocomplete="email"`. Login password: `autocomplete="current-password"`. Register email: `autocomplete="email"`. Register password: `autocomplete="new-password"`. All other sensitive fields: `autocomplete="off"`.

### Wizard navigation

- **"Save & Continue":** Runs React Hook Form `handleSubmit` for the current step's schema. On validation success → calls `saveDraftAction` with step data. On Server Action success → advances `currentStep` state. On Server Action failure → shows save-failure UI (D-03); does NOT advance step.
- **"Back":** Does NOT call any Server Action. Decrements `currentStep`. Current step's form data is already in the Zustand store from the previous save.
- **Direct step-indicator click:** Allowed only for steps ≤ `currentStep` (i.e., completed steps are navigable). Clicking a completed step indicator jumps directly to that step. Clicking a future step is disabled (indicator is non-interactive for future steps).
- **Auto-save (D-02):** Fires `saveDraftAction` on two triggers: (1) step navigation (handled by "Save & Continue"); (2) 1.5s after field blur via `useDebouncedCallback`. Debounced auto-save does NOT advance the step — it only persists the current draft silently.
- **New BIP flow:** The wizard initializes with no `bipId`. The first `saveDraftAction` call (step 1 first auto-save or step navigation) inserts a new row and returns `bipId`. All subsequent calls upsert on that `bipId`.

### Animations

All `motion` animations use `motion/react`, wrapped in `<LazyMotion features={domAnimation} strict>` — inherited rule from Phase 1.

| Animation | Duration | Easing | Trigger | Reduced-motion |
|-----------|----------|--------|---------|----------------|
| Wizard step transition | 200ms | ease-out | Step advance/back | Instant (no slide) |
| Conditional field reveal (wizard step 4) | 200ms | ease | Radio change | Instant |
| Dashboard tab content switch | 150ms | ease | Tab click | Instant |
| Save-status text change | 100ms | ease | State change | Instant |
| Auth card mount | 300ms opacity 0→1 | ease-out | Page load | Instant |
| Partner chip add | 150ms scale 0.8→1 | ease-out | Chip added | Instant |
| Partner chip remove | 100ms opacity 1→0 | ease-in | Chip removed | Instant |
| Delete/Withdraw modal | 200ms | ease-out | Button click | Instant |

### Destructive action confirmations

Confirmed via shadcn `Dialog` modal (not toast, not inline). All destructive dialogs follow this pattern:

- Title: noun + action phrase (e.g. "Delete this draft?")
- Body: personalised with BIP title in quotes.
- Two buttons stacked or side-by-side: cancel (ghost/left) + confirm (destructive/right).
- Confirm button: `bg-status-rejected text-white rounded-pill` for delete; `bg-status-pending text-white rounded-pill` for withdraw (amber, not red — withdraw is reversible).
- Cancel button: safe-action noun label — `Keep draft` for delete dialog; `Keep pending` for withdraw dialog.

### Toast

shadcn Sonner (Phase 1 baseline). Phase 2 uses the same position (bottom-center mobile, bottom-right desktop), same duration (3s confirms, 5s warnings/errors), same max 3 visible. New toast cases in Phase 2:

| Trigger | Duration | Style |
|---------|----------|-------|
| BIP submitted successfully | 5s | Default |
| Draft saved (explicit, not auto-save — only on step navigation) | 3s | Default |
| Save failed | No auto-dismiss | Error variant (red icon) |
| Session expired | 5s | Warning |
| BIP draft deleted | 3s | Default |
| BIP withdrawn | 3s | Default |
| Link copied (public StickyNav share — inherited Phase 1) | 3s | Default |

### Focus management

- Auth pages: autofocus email input on mount.
- Wizard step change: focus moves to the step title element on step advance/back.
- Dialog open (delete/withdraw/conflict): focus moves to first interactive element in dialog.
- Dialog close: focus returns to the button that triggered the dialog.
- Tab change (dashboard): focus stays on the clicked tab button.

### Loading states

- Auth submit: button disabled + `Loader2` spinner.
- Onboarding submit: same.
- Wizard "Save & Continue": button disabled + spinner. Save-status shows `Saving…`.
- Dashboard BIP list: if initial data is not available (edge case — RSC should pre-fetch), show 3 `<Skeleton>` card rows (same height as DashboardBipCard).
- University combobox: while fetching, show 3 skeleton list items inside the dropdown.

### Accessibility (Phase 2 additions to FOUN-03 baseline)

| Requirement | Implementation |
|-------------|----------------|
| Form field association | Every `<input>` / `<textarea>` / `<select>` has `id` matching its `<label>`'s `for` attribute. shadcn `<FormLabel>` + `<FormControl>` handles this automatically when `<FormField>` is used correctly. |
| Error announcement | `aria-describedby` on each input pointing to its error message element. Error message element has `role="alert"` to announce immediately on insertion. |
| Wizard step progress | Step indicator region has `role="navigation" aria-label="Wizard steps"`. Active step circle has `aria-current="step"`. |
| Dialog accessibility | shadcn `Dialog` uses Radix `Dialog.Root` — `aria-modal="true"`, focus trap, `Escape` closes (except the two-tab conflict modal — no Escape, user must choose). |
| Tab panel accessibility | shadcn `Tabs` uses Radix `Tabs.Root` — `role="tablist"`, `role="tab"`, `role="tabpanel"`, keyboard arrow navigation between tabs. |
| Combobox accessibility | shadcn `Command` inside `Popover` — `role="combobox"`, `aria-expanded`, `aria-controls`, `aria-activedescendant`. |
| Switch toggle | shadcn `Switch` uses Radix `Switch.Root` — `role="switch"`, `aria-checked`. |
| Save-status live region | Save-status indicator element has `aria-live="polite"` to announce "Saving…" / "Saved" to screen readers without interrupting. |
| Password reveal | Not implemented in Phase 1 spec; for Phase 2, add a show/hide toggle on password inputs using Lucide `Eye` / `EyeOff` icons with `aria-label="Show password"`. |

## Open Issues / Blockers This Spec Does Not Resolve

1. **University `INSERT` RLS decision:** `universities_insert_admin` currently blocks coordinator self-registration of new universities during onboarding and wizard Step 3. The planner must choose between: (a) relaxing the policy to `authenticated` for inserts from `/onboarding` + wizard, with strict server-side validation in the Server Action; or (b) using `createAdminClient` (service role) inside the `saveProfileAction` and `saveDraftAction` Server Actions for university creation. Option (b) is the safer RLS posture — isolates the bypass to a single, server-validated code path. Planner must document and implement the chosen approach.

2. **`switch` vs. `checkbox` for green travel + inclusion support:** Spec declares Switch as the preferred component for better mobile tap targets. If the planner determines Switch UX is confusing in a form context (vs. checkboxes which are more standard), substitute `checkbox` instead. Document the choice in the plan.

3. **Wizard "progress" component:** Spec mentions both step indicator dots (5 circles in header) and optionally a `progress` bar. One visual affordance is sufficient. Planner should choose step indicator dots (simpler, aligns with spec) and omit the `progress` bar component unless user explicitly requests it.

4. **`onAuthStateChange` + localStorage backup (SUBM-07):** The session expiry recovery path requires a `useEffect` in `<BipSubmissionWizard>` that subscribes to `supabase.auth.onAuthStateChange`. When `SIGNED_OUT` fires, the wizard saves the Zustand store to `localStorage["biphub:draft"]` (JSON), shows the session-expired toast, and redirects to `/login?redirect=/dashboard/bips/new`. On return to the wizard after re-login, check `localStorage["biphub:draft"]` and restore the Zustand store. The UI confirmation for restore: a non-blocking banner at the top of Step 1 — `We restored your unsaved draft from before your session expired.` with a `Dismiss` text link. This restore logic is a coordinator surface edge case; the planner must spec the Server Action restart path precisely.
