    ## Section 1 — Quick start

    Numbered steps for a fresh clone:
    1. Install prerequisites: Node 22 LTS, npm, Supabase CLI (`npm i -g supabase` or scoop / brew).
    2. Clone the repository and `cd biphub`.
    3. `cp .env.example .env.local`.
    4. `supabase start` (boots local Postgres + Auth + Storage in Docker).
    5. Run `npx supabase status` and copy the printed `anon` and `service_role` keys into `.env.local` — `NEXT_PUBLIC_SUPABASE_ANON_KEY` and `SUPABASE_SERVICE_ROLE_KEY` respectively.
    6. `npm install`.
    7. `npm run db:reset` to apply migrations and load `supabase/seed.sql` (20 BIPs).
    8. `npm run dev` and open http://localhost:3000.

    Note about Supabase CLI ≥2.98: the new `sb_publishable_*` / `sb_secret_*` keys must be re-pulled from `npx supabase status` after each `supabase start` — legacy JWT-style keys no longer authenticate against PostgREST. This bit a previous contributor; documented here verbatim.

    ## Section 4 — Code conventions

    A concise list (use checkboxes for visual scanability):

    - [ ] Use `getClaims()` server-side; NEVER `getSession()`. The unvalidated session reader does not validate JWT signatures.
    - [ ] `await cookies()` in every Supabase server client factory. Next.js 15 made this async — synchronous calls silently never set the cookie.
    - [ ] Tailwind class names must be literal strings. The v4 static scanner cannot resolve template literals. For variant-based classes, build a lookup object of complete class strings.
    - [ ] `motion` package, never `framer-motion` (deprecated alias). Wrap motion components in `LazyMotion` to avoid the always-loaded 34KB bundle.
    - [ ] Zod v3 + `@hookform/resolvers` v3.x — `@hookform/resolvers` does not yet ship a TypeScript-stable Zod 4 adapter.
    - [ ] Never call `createAdminClient` outside `app/(admin)/` or `lib/supabase/admin.ts`. The ESLint rule `no-restricted-imports` enforces this; do not disable it.
    - [ ] Every public-route page must inherit the public Footer with the disclaimer "Independent project — not affiliated with the European Commission".
    - [ ] No analytics scripts. No Plausible, no GA, no Vercel Analytics. If you need them, also add a consent banner — talk to the maintainers first. (FOUN-05 posture is "ship zero trackers"; revisit only with product-market-fit evidence.)

    ## Section 5 — Database changes

    Migration filename convention: `supabase/migrations/000NN_short_description.sql`. The number is the next sequential integer; padding to 5 digits.

    After writing a migration, run:
    ```
    npm run db:reset     # tears down and re-applies all migrations + seed
    npm run db:types     # regenerates lib/supabase/database.types.ts
    ```

    RLS policy template (D-25, copied from migration 00006):

    Every `UPDATE` policy MUST have both `USING` (which rows can be targeted) and `WITH CHECK` (which post-image rows are accepted). Without `WITH CHECK`, a coordinator could reassign `created_by` to another user. PITFALLS.md Pitfall 5 covers this.

    Paste this template inline as a code block:
    ```sql
    create policy "table_update_own"
      on public.table_name for update
      to authenticated
      using (
        (select auth.uid()) = owner_id
        and status in ('eligible_pre_states')
      )
      with check (
        (select auth.uid()) = owner_id
        and status in ('eligible_post_states')
      );
    ```

    Hard constraints:

    - NO `continue-on-error: true` — a finding MUST block the merge.
    - Workflow runs on `push` to `main` (catches direct pushes by maintainers) AND `pull_request` to `main` (catches contributor PRs).
