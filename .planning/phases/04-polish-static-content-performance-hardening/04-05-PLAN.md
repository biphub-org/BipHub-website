    ```sql
    -- 00013_delete_my_account.sql
    -- FOUN-07 / Phase 4 D-09: coordinator-driven account deletion (GDPR Art 17).
    --
    -- This RPC runs atomically (single transaction) under SECURITY DEFINER:
    --   1. Anonymize approved BIPs owned by the calling user (contact_name='—',
    --      contact_email=NULL). RLS would not allow this UPDATE for the
    --      coordinator (bips_update_own_editable blocks status='approved');
    --      DEFINER privilege is the controlled exception.
    --   2. Hard-delete bips owned by the user with status IN ('draft','pending','rejected').
    --   3. Delete the auth.users row. The FK cascade chain then fires:
    --        - profiles.id ON DELETE CASCADE → profiles row removed
    --        - remaining bips.created_by ON DELETE SET NULL → approved BIPs survive anonymized
    --        - bip_status_history.actor_id ON DELETE SET NULL → audit log preserved
    --
    -- Security properties:
    --   - SECURITY DEFINER with set search_path = public, auth, pg_temp prevents
    --     search-path-injection privilege escalation (PITFALLS.md general SQL hardening).
    --   - Function reads auth.uid() internally — NO user_id parameter. The caller
    --     cannot delete another user's account regardless of input.
    --   - Owner = postgres (the supabase admin role). EXECUTE granted to
    --     'authenticated' only; revoked from public and anon.
    --   - Atomic: any failure mid-flight rolls back the entire chain. There is
    --     no half-deleted state.
    ```

    The RPC body:
    ```sql
    create or replace function public.delete_my_account()
    returns void
    language plpgsql
    security definer
    set search_path = public, auth, pg_temp
    as $$
    declare
      caller uuid;
    begin
      caller := auth.uid();
      if caller is null then
        raise exception 'delete_my_account: not authenticated' using errcode = '42501';
      end if;

      -- Step 1: anonymize approved BIPs (preserve directory content; remove PII).
      update public.bips
        set contact_name  = '—',
            contact_email = null
        where created_by = caller
          and status = 'approved';

      -- Step 2: hard-delete draft/pending/rejected BIPs (cascade removes partner rows).
      delete from public.bips
        where created_by = caller
          and status in ('draft', 'pending', 'rejected');

      -- Step 3: delete the auth.users row. Cascades remove profiles row;
      -- bips.created_by / bip_status_history.actor_id are set NULL.
      delete from auth.users where id = caller;
    end;
    $$;

    -- Lock down execution.
    revoke all on function public.delete_my_account() from public, anon;
    grant execute on function public.delete_my_account() to authenticated;

    -- Owner must be postgres (the supabase superuser role). The CLI loads
    -- migrations as postgres by default, so the function is created with the
    -- correct owner automatically; no explicit ALTER FUNCTION OWNER required.
    -- (Verified via PITFALLS.md security-definer ownership pattern.)
    ```

    Hard constraints:
    - `auth.uid()` MUST be read inside the function body, NEVER accepted as a parameter — this is the cross-user-deletion mitigation (T-04-14 in threat model).
    - `set search_path = public, auth, pg_temp` is REQUIRED on the function declaration — without this, a malicious extension installed in another schema could shadow `auth.uid()` and elevate.
    - `auth, pg_temp` are included in search_path because the function references `auth.users`; `pg_temp` is a defensive habit recommended by postgres docs.
    - `language plpgsql` is required (not `sql`) because the body has multiple statements + a conditional raise.
    - EXECUTE grant is to `authenticated` only — NOT to `anon` (anon has no uid()), NOT to `public` (would leak via grants).
    - The function MUST NOT return anything that leaks user data (return `void`).

    Implements: FOUN-07 + D-09.

    ```ts
    export async function deleteAccountAction(formData: FormData): Promise<never> {
      const supabase = await createClient()

      // 1. Verify the caller is authenticated and resolve their email.
      const { data: claimsData, error: claimsErr } = await supabase.auth.getClaims()
      if (claimsErr || !claimsData?.claims?.sub) {
        redirect('/login')
      }
      const claims = claimsData.claims
      const sessionEmail = typeof claims.email === 'string' ? claims.email : ''
      const userId = claims.sub

      // 2. Validate the typed-email confirmation (defence in depth — the modal
      //    button is disabled until typed === claims.email, but the user could
      //    bypass the disabled state via DevTools).
      const parsed = DeleteAccountSchema.safeParse({
        typedEmail: formData.get('typedEmail'),
      })
      if (!parsed.success) {
        throw new Error('Type your account email to confirm.')
      }
      if (parsed.data.typedEmail.toLowerCase().trim() !== sessionEmail.toLowerCase().trim()) {
        throw new Error('Typed email does not match your account email.')
      }

      // 3. BEFORE deletion: collect slugs of approved BIPs we are about to
      //    anonymize. We need these for revalidatePath() AFTER the RPC fires.
      const { data: approvedSlugs } = await supabase
        .from('bips')
        .select('slug')
        .eq('created_by', userId)
        .eq('status', 'approved')

      // 4. Fire the SECURITY DEFINER RPC. It atomically:
      //    - anonymizes approved BIPs (contact_name='—', contact_email=NULL)
      //    - deletes draft/pending/rejected BIPs
      //    - deletes the auth.users row (cascades remove profiles)
      const { error: rpcErr } = await supabase.rpc('delete_my_account')
      if (rpcErr) {
        throw new Error(`Account deletion failed: ${rpcErr.message}`)
      }

      // 5. Sign out the now-deleted session (clears the auth cookie).
      await supabase.auth.signOut()

      // 6. Bust ISR caches for /bips (counts changed) and each anonymized slug.
      revalidatePath('/bips')
      for (const row of approvedSlugs ?? []) {
        if (row?.slug) revalidatePath(`/bip/${row.slug}`)
      }
      revalidatePath('/')

      // 7. Redirect to homepage with the deletion-toast trigger param.
      redirect('/?deleted=1')
    }
    ```

    Hard constraints:
    - `'use server'` directive at top of file (required for Server Actions).
    - `await cookies()` is invoked indirectly through `createClient()` — verify the existing server.ts factory does this; if not, abort and fix server.ts first.
    - `getClaims()`, NEVER `getSession()` — CLAUDE.md lock.
    - NO `createAdminClient` — the SECURITY DEFINER RPC is the controlled escalation; the action runs as the coordinator.
    - The `typedEmail` comparison MUST be case-insensitive AND trimmed — emails are case-insensitive per RFC 5321.
    - Slug collection MUST happen BEFORE the RPC fires — after deletion, the rows are anonymized but `created_by` is NULL, so we couldn't filter them.
    - DO NOT send any confirmation email per D-10 (the email account is destroyed; sending to it is pointless).
    - DO NOT swallow RPC errors silently — re-throw with the Postgres error message so the modal can display it (the modal renders the thrown error in a toast).

| T-04-14 | E (Elevation of privilege) | Cross-user deletion via crafted RPC call | mitigate | `delete_my_account()` takes ZERO parameters; the user_id is read internally via `auth.uid()`. | T-04-16 | E (Elevation of privilege) | RPC owner privilege drift | mitigate | Migration loaded by Supabase CLI as `postgres` (superuser); function is owned by postgres automatically. No `ALTER FUNCTION OWNER` needed in v1. CONTRIBUTING.md notes this in the DB-changes section. |
| T-04-17 | S (Spoofing) | Confirmation bypass — DevTools enabling disabled button | mitigate | Server Action re-validates typed-email server-side (defense in depth). UI disabled state is UX, not security. Acceptance criteria covers both layers. |
